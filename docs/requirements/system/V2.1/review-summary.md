## 新たな致命的問題

### D1: 比較指標の単位未定義（ランキングの信頼性）
**指摘元**: Codex

`kip7`（KIP-7トークン）と`native`（ネイティブKAIA送金）を同じ`weekly_token_distributed`で集計しているが、トークン種類・桁数・単位が異なる場合、横比較が成立しない。ランキングが誤誘導になるリスク。

**修正案**:
- `raw_amount`（トークン別生値）と `normalized_value_kaia`（KAIA換算値）を分離
- ランキングは換算可能データのみ対象
- 換算不能ゲームは「比較対象外」に明示

### D2: reward_type別のmetricsスキーマ矛盾
**指摘元**: Codex

SRS本文では`nft`は`active_wallets_7d`のみ対象だが、`metrics`型は全フィールド必須（`weekly_token_distributed`, `reward_concentration`等）。NFTゲームで`weekly_token_distributed: 0`のダミー値が混入する。

**修正案**:
- 各メトリクスを個別optionalにする
- `metric_availability`フラグを追加し、計測可能な指標を明示
- または判別共用体で`reward_type`ごとに異なるmetrics型を定義

## 重要な指摘

### R1: RPCヘルスチェック条件の強化
**指摘元**: Codex

レスポンス1秒超だけでは、レスポンスは速いが古いブロックを返すノードを検知できない。

**推奨**: `chainId`一致確認、最新ブロック遅延（≤2ブロック）、連続失敗時クールダウンを追加。

### R2: BRD/SRS間の品質ゲート記述不一致
**指摘元**: Codex

BRDは4要素（文字数+重複率+オリジナルデータ率+ファクトチェック）、SRSは5要素（+NGワード）。

**推奨**: SRSの5段階定義に統一し、noindex解除条件も明文化。

### R3: Phase A-0ゲートの強化
**指摘元**: Codex + Claude

「≥5本」は最低ラインだが、上位流入ゲームのカバレッジ基準がない。計測可能ゲームが5本あっても、人気ゲームが含まれなければランキングの価値が低い。

**推奨**: 本数基準に加え、「プレイ人数上位20ゲームの60%以上を計測可能」というカバレッジ条件を追加。

### R4: data_confidenceのUX強化
**指摘元**: Codex

high/medium/lowだけでは理由がわからない。

**推奨**: `confidence_reason`フィールド（例: `contract_verified`, `rpc_degraded`, `partial_nft`）をAPIとUIに追加。

### R5: TF-IDF閾値の調整
**指摘元**: Codex + Claude

80%はパラフレーズ重複を通しやすい。日本語の形態素解析が必要。

**推奨**: 初期閾値を70-75%に設定。n-gramベースのJaccard類似度（形態素解析不要）を代替候補として検討。

### R6: トークンフロー分類の拡張
**指摘元**: Codex

2分類（取引所/その他）はMVPとしては可だが、`unknown/unclassified`を別建てし、分類辞書のバージョン管理を入れるべき。

### R7: 予備費の発動基準
**指摘元**: Codex

予備費¥2,500の使途基準が未定義。

**推奨**: 「API使用率80%超で月末まで予備費投入」「RPC失敗3日連続で有料プラン移行」等のルールを定義。

### R8: I1-I9対応のトレーサビリティ
**指摘元**: Codex

V2.0のI1-I9がV2.1のどの節で対応されたか追跡できない。

**推奨**: `I# → V2.1節 → ステータス` の対応表を作成。

### R9: Phase A-0の調査手順定義
**指摘元**: Codex

手順がないと1ゲーム30分以上かかり、20ゲーム×30分=10時間で2日では終わらない。

**推奨**: 1ゲームあたり20-30分の上限、判定テンプレート（KaiaScan検索→トランザクション確認→reward_type判定フロー）を事前定義。

### R10: 差別化の参入障壁
**指摘元**: Codex

オンチェーンデータの取得手法自体は模倣可能。

**推奨**: 「蓄積された時系列データ」「再現可能な算出ログ」「更新速度（速報性）」をモートとして設計に明記。先行者の時系列データは後発には再現不可能。

## 改善提案

### P1: API metaにブロック範囲とエンドポイント情報を追加
`source_block_range`と`endpoint_used`を含めることで、データの再現性・監査可能性を向上。

### P2: 低信頼データ時のフェイルセーフ表示
RPC障害等でデータ品質が低下した場合、ランキング更新を停止し「前週データ」を表示する機能。

### P3: 一次データ引用の最小数ルール
品質ゲートに「一次データ引用箇所 ≥ 2箇所」を追加し、データソースセクション以外の本文内にも引用を求める。

## 次のアクション

D1-D2は実装時に解決可能な技術的問題であり、要件定義の方向性変更は不要。以下の対応で実装に進める:

1. **SRSのmetricsスキーマを修正**: 各フィールドをoptionalにし、`metric_availability`を追加
2. **ランキングの単位定義を追加**: Phase A-0の調査で単位の統一性を確認し、必要なら`normalized_value`を導入
3. **RPCヘルスチェック条件を拡張**: ブロック遅延チェックを追加
4. **品質ゲートをSRS基準（5段階）に統一**
5. **Phase A-0の調査手順テンプレートを作成**
6. **上記を反映して実装着手**（BRD/SRSの再改訂は不要、実装時に対応）

**判断**: D1-D2はスキーマ設計の詳細であり、Phase A-0の調査結果を見て最終化する方が効率的。要件定義としてはこのままGoとし、実装フェーズで確定させる。
