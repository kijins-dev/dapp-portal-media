## BRD V2.0 — Dapp Portal攻略ガイド オリジナリティ自動化基盤

### 背景
LINE Dapp Portal（Mini Dapp）の全ゲーム攻略記事を自動生成するメディアサイト。V1.xで基盤構築済み（20記事公開、SEO内部最適化完了）。V2.0では**AI全自動でオリジナリティを確保**し、SEO・LLMOで日本語圏1位を目指す。

### 課題
- LLM生成テキストのみで独自の一次データがない
- 競合が同じLLMで記事を生成すれば差別化が消失
- E-E-ATの「Experience」要素が欠如
- LLM引用される差別化データがない

### 戦略: 4つの柱

**柱1: オンチェーン・インテリジェンス（最重要）**
- Kaiaブロックチェーン公開RPCノードから、ゲーム別のトークン配布量・アクティブウォレット数・報酬集中度（Gini係数）・トークンフローを自動取得
- 週間ランキングページ、公開JSON APIとして配信
- コスト: 無料（公開RPCノード）、bot検出リスク: ゼロ

**柱2: 非操作クライアント観測**
- Playwrightでゲーム起動→初期画面スクショ自動撮影（ゲーム操作はしない）
- UI変更検出（Patch Diff Radar）: 前回との画像差分 + アセットhash比較
- HAR通信ログ収集、プレイ人数時系列トラッキング

**柱3: 需要駆動型コンテンツ生成**
- GSC Search Analytics APIから未充足クエリ（インプレッションあるがコンテンツなし）を自動抽出
- 比較記事・FAQ記事を自動生成、既存記事の自動リライト

**柱4: LLM引用最適化（LLMO）**
- 全記事に構造化FAQ（schema.org FAQPage）
- 公開JSONエンドポイント（更新日時・計測手法・信頼度付き）
- robots.txtでOAI-SearchBot/PerplexityBot/GoogleOther明示許可
- llms.txt新規作成

### 制約
- 全自動（人間介入は月1回モニタリングのみ）
- 月額5,000円以内
- ゲーム自動操作は行わない（Canvas/WebGL問題 + 利用規約リスク）

### やらないこと
- ゲーム自動プレイ（技術・法務リスク高）
- SNSデータ主軸利用（補助指標のみ）
- 動画コンテンツ、有料コンテンツ、多言語対応

## SRS V2.0（要約）

### オンチェーンデータ収集
- ethers.js + Kaia RPC（`https://public-en.node.kaia.io`）
- KIP-7 Token Transferイベント集計、ブロックタイムスタンプベース
- 週次GitHub Actions cron、レート制限対策あり
- 出力: `data/onchain/game-metrics.json`, `ranking.json`, `history/{slug}.json`

### 指標
| 指標 | 算出方法 |
|------|---------|
| 週間KAIA配布量 | Transferイベント合計（コントラクト→外部） |
| アクティブウォレット数 | 7日間ユニークfromアドレス |
| 報酬集中度 | 上位10%ウォレットの占有率 |
| トークンフロー | Transfer先分類（取引所/再投資/保持） |

### 観測基盤
- Playwright: 起動→5秒待機→スクショ（WebP、1280px）
- pixelmatch: 画像差分5%以上で「更新あり」
- HAR: APIエンドポイント一覧・エラー率抽出
- プレイ人数: 既存スクレイパー日次記録→時系列グラフ

### 需要駆動
- GSC API: 過去28日、インプレッション>10, CTR<3%, 対応記事なし
- 分類→テンプレート選択→Claude API生成→品質ゲート→配置

### 公開データAPI
```json
{
  "meta": {
    "generated_at": "ISO8601",
    "methodology": "計測手法の説明",
    "confidence": 0.95
  },
  "games": [...]
}
```

### 非機能要件
- コスト: 月額¥2,500以内（Claude Haiku中心）
- オンチェーン収集: 76ゲーム30分以内
- RPCフォールバック: QuickNode無料枠
- 3回連続失敗→自動停止+アラート

### 実装フェーズ
- Phase A (Week 1-2): オンチェーン基盤
- Phase B (Week 2-3): 観測基盤
- Phase C (Week 3-4): 需要駆動+LLMO
- Phase D (Week 4-5): 統合・自動化

## レビュー観点

### 要求定義の観点
1. **目的の明確性**: 「AI全自動でオリジナリティを確保し日本語圏1位」は達成可能か？
2. **ユーザーストーリーの網羅性**: プレイヤーの主要ニーズを捉えているか？
3. **スコープの妥当性**: 4つの柱の優先度は適切か？やらないことは正しいか？
4. **ビジネスゴールの測定可能性**: KPIは現実的か？

### 要件定義の観点
1. **要求との整合性**: BRDの戦略が技術仕様に正しく落とし込まれているか？
2. **データ設計**: オンチェーンデータの取得方法は実現可能か？Kaiaの技術的制約は？
3. **技術適合性**: ethers.js + 公開RPCノードで月5,000円以内に収まるか？
4. **UX**: ランキングページ・データAPIの設計は利用者にとって有用か？
5. **運用面**: 週次cronの失敗対策は十分か？ゲームの追加・削除への対応は？
6. **セキュリティ**: 公開JSONにプライバシー上の問題はないか？
7. **矛盾・不整合**: BRDとSRS間、またはSRS内部で矛盾はないか？

### 追加で特に見てほしい点
1. **オンチェーンデータの実現可能性**: LINE Dapp Portalのゲームがすべて個別のスマートコントラクトを持っているとは限らない。KAIAトークンの配布方法がゲームごとに異なる場合、統一的な計測は可能か？
2. **Scaled Content Abuse回避**: この規模のAI生成コンテンツはGoogleのポリシーに抵触しないか？品質ゲートは十分か？
3. **差別化の持続性**: 競合がこの手法を模倣した場合のモート（参入障壁）は何か？
4. **コスト見積もりの妥当性**: Claude API + GitHub Actions + RPC、月5,000円は現実的か？
