# V2.0 レビュー統合結果

> レビュー実施: 2026-03-10
> レビュアー: Codex (GPT-5.3) / Claude (Opus 4.6) / Gemini (エラーで参加不可)

## 総合評価: Conditional Go

方向性は正しいが、3つの致命的な問題を解決してからPhase Aに進むべき。

---

## 致命的な問題（実装をブロックするもの）

### C1: 計測モデルの前提崩壊 — 報酬タイプの多様性
**指摘元**: Codex + Claude

SRSはKIP-7 Token Transferイベントのみを計測対象としているが、LINE Dapp Portalのゲーム報酬は以下の4タイプが混在する:

| 報酬タイプ | 例 | オンチェーン追跡 |
|-----------|-----|----------------|
| KIP-7トークン（KAIA） | 一部ゲーム | 可能 |
| ネイティブKAIA送金 | 一部ゲーム | 可能（別メソッド） |
| NFT（KIP-17） | コレクション系 | 可能（別イベント） |
| オフチェーンポイント | 多くのゲーム | **不可能** |

**影響**: 76ゲーム中、KIP-7 Transferで追跡可能なゲームが少数の可能性。統一指標としての「週間KAIA配布量」が機能しない。

**修正案**:
1. `config/contracts.yaml` に `reward_type` フィールドを追加（`kip7` / `native` / `nft` / `offchain` / `unknown`）
2. Phase A開始前に上位20ゲームの報酬タイプを手動調査（KaiaScanで確認）
3. オフチェーン報酬ゲームは `metrics: null` として扱い、プレイ人数・スクショ等の他指標で補完
4. ランキングページは「オンチェーン計測可能ゲーム」と明示

### C2: 公開RPCノードのSLA不在
**指摘元**: Codex

`https://public-en.node.kaia.io` は開発・テスト用であり、本番環境向けのSLAが存在しない。週次cronが安定稼働する保証がない。

**修正案**:
1. QuickNodeフォールバックを「バックアップ」ではなく「プライマリの一つ」に格上げ
2. RPCエンドポイントの優先度リスト化: 公開RPC → QuickNode無料枠 → Alchemy無料枠
3. ヘルスチェック（`eth_blockNumber`）を各実行前に実施し、レスポンス1秒以上なら次のエンドポイントへ
4. 月次のRPC可用性ログを記録し、問題が頻発すればAlchemy有料枠（$0/月〜）を検討

### C3: Scaled Content Abuse回避の品質ゲート不足
**指摘元**: Codex + Claude

品質ゲートが「文字数 < 1500字 or NGワード検出 → noindex」のみ。Googleの「Scaled Content Abuse」ポリシーに対して不十分。

**修正案**:
1. 品質ゲートの強化:
   - 文字数: 1,500字未満 → noindex（現行維持）
   - 重複率: 既存記事との類似度 > 80%（TF-IDFベース） → noindex
   - オリジナルデータ率: オンチェーンデータ・スクショ等の一次データ参照がゼロ → noindex
   - ファクトチェック: 記事内の数値がdata/*.jsonの値と不一致 → 生成やり直し
2. 生成ペースの制御: 1日最大5記事（急激な大量公開を避ける）
3. 各記事に「データソース」セクションを必須化（オンチェーンデータの引用元を明記）

## 重要な指摘（対応すべきもの）

### I1: BRDとSRSのコスト矛盾
**指摘元**: Codex

BRDは「月額5,000円以内」、SRSは「月額¥2,500以内」。どちらが正しいか統一が必要。

**推奨**: SRSの¥2,500をAPI費用のみとし、BRDの¥5,000を「API + 予備費（QuickNode有料化時等）」として両立させる。

### I2: アクティブウォレットの定義曖昧
**指摘元**: Codex

SRSでは「ユニークfromアドレス」だが、to（受取）側もアクティブユーザー。ゲームコントラクトへのトランザクション送信者（from）に限定すべきか、報酬受取者（to）も含むべきか。

**推奨**: ゲームコントラクトとの相互作用アドレス（from + to）をアクティブウォレットとし、指標定義をSRSに明記。

### I3: コントラクトアドレス特定の実現可能性
**指摘元**: Claude

「KaiaScanで手動特定」とあるが、76ゲーム全てが個別のスマートコントラクトを持つとは限らない。Dapp Portalの中間コントラクト経由で報酬配布される場合、ゲーム別の切り分けが困難。

**推奨**: Phase A開始時に上位10ゲームで調査し、コントラクト構造のパターンを把握してから全体設計を確定する。

### I4: GSCデータの遅延問題
**指摘元**: Claude

GSC Search Analytics APIのデータは2-3日遅延する。「インプレッション > 10 かつ CTR < 3%」の条件で抽出する未充足クエリが、すでに対応済みの可能性がある。

**推奨**: 抽出時に既存記事のslug・タイトルとのマッチングを行い、重複生成を防止するロジックを追加。

### I5: GitHub Actions実行時間の予算
**指摘元**: Claude

無料枠2,000分/月に対し、週次実行の見積もり:
- オンチェーン収集: 30分 × 4週 = 120分
- スクリーンショット: 60分 × 4週 = 240分
- 記事生成・リライト: 推定30分 × 4週 = 120分
- サイトビルド・デプロイ: 5分 × 4週 = 20分
- 合計: 約500分/月

余裕はあるが、デバッグ・リトライ・日次タスク（プレイ人数）を含めると1,000分超の可能性。

**推奨**: 日次タスク（プレイ人数トラッキング）の実行時間も見積もりに含め、月間予算を明記。

### I6: HAR通信ログのセキュリティ
**指摘元**: Codex

HARファイルにはCookie、認証トークン、ユーザーエージェント等のセンシティブ情報が含まれる可能性。公開リポジトリにコミットするとPII漏洩リスク。

**推奨**: HAR解析結果（`har-summary/{slug}.json`）のみ保存し、生HARファイルは処理後に削除。Cookie/Authorization headerは解析時にマスク。

### I7: トークンフロー分類の精度
**指摘元**: Codex

Transfer先を「取引所/再投資/保持」に分類するには、既知の取引所アドレスリストが必要。Kaiaエコシステムの取引所アドレスが網羅できるか不明。

**推奨**: 初期は「取引所（既知アドレスリスト）/ その他」の2分類でスタートし、段階的に精度を上げる。

### I8: Gini係数の算出方法
**指摘元**: Codex

SRSでは「上位10%ウォレットの占有率」を「Gini係数」と呼んでいるが、これは正確にはGini係数ではない（Gini係数はローレンツ曲線から算出）。

**推奨**: 指標名を「報酬集中度（上位10%占有率）」に変更するか、正式なGini係数計算を実装。

### I9: 画像差分検出の閾値
**指摘元**: Codex

pixelmatchの5%閾値は、広告バナーの変更やアニメーションで誤検知する可能性。

**推奨**: 閾値を調整可能にし、初期運用で最適値を探る。広告領域のマスク機能も検討。

## 改善提案（あると良いもの）

### P1: ゲーム廃止・新規追加の自動検出
Dapp Portalのゲーム一覧ページを定期スクレイピングし、新規ゲーム追加・既存ゲーム削除を自動検出→記事生成キューに追加/アーカイブ化。

### P2: データの信頼度スコア
オンチェーン計測可能ゲーム（信頼度: 高）、プレイ人数のみ（信頼度: 中）、情報なし（信頼度: 低）の3段階を各ゲームに付与し、公開JSONにも反映。

### P3: llms.txtの段階的拡充
初期はサイト概要+データAPI一覧のみ。引用状況をモニタリングし、効果があればゲーム別のサマリーも追加。

---

## 良い点

1. **4つの柱の戦略構造**: オンチェーンデータという模倣困難な差別化要素を中核に据えた設計は正しい
2. **ゲーム自動操作の明確な除外**: 技術・法務リスクを正しく評価し、スコープから除外した判断は適切
3. **公開JSON API + LLMO**: データの機械可読性を高め、LLM引用を狙う戦略は先進的
4. **コスト意識**: 無料枠中心の設計で月額5,000円以内は現実的な目標
5. **段階的なフェーズ分け**: 5週間のフェーズ分けで、最重要の柱1から着手する優先度設計が適切
6. **needs-based content generation**: GSCデータに基づく需要駆動型コンテンツ生成は、スパム判定回避にも有効
